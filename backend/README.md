# Backend

Phase 0 fixes the provider-neutral AI contract and test assets. Phase 1 adds
the FastAPI/PostgreSQL foundation. Phase 2 adds bounded RSS collection, a
PostgreSQL durable queue, and independent scheduler/worker processes.

With `uv` installed:

```sh
uv sync --frozen --dev
uv run --frozen pytest
```

The tests are offline: they use synthetic provider payloads and a short,
synthetic Tuoi Tre-shaped RSS fixture.

## Local stack

1. Copy `.env.example` to `.env` at the repository root.
2. Replace the PostgreSQL password and set `ADMIN_PASSWORD_HASH` to an
   Argon2id hash generated by the command documented in that file.
3. Run `docker compose up --build`.

The API listens only on `127.0.0.1:8080`. PostgreSQL has no host port. The
`migrate` service applies Alembic and bootstraps the single administrator
before the API starts. Schema creation never happens in application startup.
Bootstrap is create-only and idempotent: restarting Compose never rotates the
password. Operators rotate explicitly with
`python -m vietnam_calendar.bootstrap --rotate-password`; rotation revokes all
active sessions in the same transaction. Expired/revoked sessions can be
removed with `--prune-sessions`.

The local single-process API rate-limits failed logins in memory (five per
minute per normalized client IP), evicts stale entries, and caps tracked keys
at 4096. Rotating usernames therefore does not bypass the limit. A
multi-process deployment must replace this with
a shared limiter before exposure beyond localhost.

Phase 1 endpoints:

- `GET /healthz` — process liveness, intentionally no database access
- `GET /readyz` — database connectivity and Alembic-head readiness
- `POST /api/v1/auth/login` — sets an HttpOnly SameSite=Strict session cookie
- `POST /api/v1/auth/logout` — requires `X-CSRF-Token`
- `GET /api/v1/auth/me` — authenticated administrator
- `GET /api/v1/feeds` — authenticated feed list

Phase 2 endpoints (all administrator-only):

- `POST /api/v1/feeds/{feed_id}/fetch` — requires `X-CSRF-Token`; returns a durable job ID with HTTP 202
- `GET /api/v1/jobs?limit=50` — recent queue state and safe failure codes
- `GET /api/v1/fetch-runs?limit=50` — RSS provenance and insert/update counts

The scheduler checks due feeds once per UTC minute. The worker claims one job
using `FOR UPDATE SKIP LOCKED`, commits the lease before network access, then
stores the raw response compressed in `fetch_runs`. RSS tests use only fixed
fixtures and HTTPX `MockTransport`; they never depend on the live publisher.
