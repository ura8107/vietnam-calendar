# RSS 日誌カレンダー 要件定義書

## 1. 文書情報

- 対象プロジェクト: Vietnam Calendar
- 文書目的: RSS からベトナムに関する情報を自動収集し、重要な出来事を日誌としてカレンダー形式で確認できるサービスの要件を定義する
- 想定リリース: MVP（個人利用〜小規模な一般公開）
- 対象言語: 画面は日本語。原文のタイトル・本文・URLは保持する

### 1.1 確定済みプロダクト判断

- 初期 RSS: `https://news.tuoitre.vn/home.rss`（Tuoi Tre News）
- 利用範囲: 当面は一般公開せず、単一運用者によるローカルまたは閉域利用とする
- AI 実行方式: 外部 API とローカルモデルの両方に対応し、設定で切り替え可能にする
- 重要度基準: 初期値を固定せず、運用者との対話と記事レビューから段階的に形成する
- 公開判定: 初期はすべて人間が確認し、十分な評価データと合意済み基準が得られた処理だけを段階的に自動化する
- 日付基準: 出来事の発生日を優先し、特定できない場合は記事公開日を使う

## 2. 背景と課題

複数のニュースサイトや公的機関の更新情報を継続的に追うには、サイトごとの巡回、重複記事の整理、重要な出来事の選別が必要になる。現在の Vietnam Calendar は静的な案内画面のみで、情報の収集・保存・検索機能を持たない。

本機能では RSS を定期収集し、記事を日付単位で整理する。単なる記事一覧ではなく、同じ出来事をまとめ、重要度と根拠を付けた「日誌」を生成し、カレンダーから過去の動きを振り返れる状態を目指す。

## 3. ゴールと成功指標

### 3.1 ゴール

1. 登録済み RSS を人手なしで定期取得できる
2. ベトナムに関係する重要な出来事を日単位で把握できる
3. 同一内容の記事をまとめ、情報過多を抑えられる
4. 出典へ戻って内容を検証できる
5. 収集失敗や判定結果を運用者が確認・修正できる

### 3.2 MVP 成功指標

- 有効なフィードの取得成功率: 月間 95% 以上（配信元起因の障害を除く）
- 新着記事の反映: 配信から原則 60 分以内
- 同一記事の二重登録率: 1% 未満
- 重要な出来事の各表示に、最低 1 件の出典 URL があること: 100%
- カレンダーの月表示: 通常回線で主要コンテンツを 3 秒以内に表示
- 日次処理の失敗を、次回実行までに運用者が識別できること

## 4. スコープ

### 4.1 MVP に含める

- RSS/Atom フィードの登録、有効化、無効化
- 定期取得と手動再取得
- 記事の正規化、重複排除、ベトナム関連度の判定
- 同一出来事に関する複数記事のグループ化
- 重要度の算出と、重要な出来事の日誌化
- 月・日カレンダー表示
- 日別の出来事一覧と詳細表示
- キーワード・期間・カテゴリ・重要度による絞り込み
- 原文タイトル、出典、公開日時、原文 URL の表示
- 収集状況とエラーを確認する管理機能
- 手動での重要度変更、非表示、出来事の統合・分割

### 4.2 MVP に含めない

- RSS がないサイトのスクレイピング
- SNS の直接収集
- 有料記事本文の取得や転載
- ユーザーごとの推薦、コメント、共同編集
- ネイティブアプリ、プッシュ通知、メール配信
- 完全自動の事実認定や真偽判定
- 多言語 UI（原文保持は行う）

## 5. 想定利用者

### 閲覧者

- ベトナムの政治、経済、社会、災害、文化などの動向を日付から振り返りたい
- 出来事の概要を短時間で把握し、必要に応じて原文を確認したい

### 運用者

- RSS フィードを管理したい
- 取得エラーや未処理記事を確認したい
- 自動判定の誤りを修正したい

MVP は一般公開せず、localhost またはアクセス制限されたネットワークで利用する。外部ネットワークから到達可能にする場合は、閲覧画面を含む全画面を認証必須とする。管理操作は利用形態にかかわらず認証済み運用者だけに許可する。

## 6. 主要ユースケース

1. 閲覧者が月を選ぶと、出来事のある日と重要度がカレンダー上に表示される
2. 日付を選ぶと、その日に発生または報道された重要な出来事を確認できる
3. 出来事を開くと、要約、カテゴリ、重要度の根拠、関連する出典を確認できる
4. 閲覧者がキーワード、期間、カテゴリで過去の出来事を探す
5. 運用者が RSS URL を登録し、接続テスト後に収集を有効化する
6. 運用者が収集失敗を確認し、再実行またはフィードを無効化する
7. 運用者が誤って統合された出来事を分割し、重要度や公開状態を修正する

## 7. 機能要件

### FR-01 RSS フィード管理

- 運用者は名称、RSS/Atom URL、媒体名、言語、カテゴリ、取得間隔を登録・編集できる
- 登録時に URL 到達性とフィード形式を検証し、結果を表示する
- フィード単位で有効・無効を切り替えられる
- 最終成功日時、最終失敗日時、連続失敗回数、直近エラーを確認できる
- 同一 URL の重複登録を防止する
- 初期データとして次のフィードを登録し、MVP の動作確認に使用する

| 項目 | 初期値 |
|---|---|
| 名称 | Tuoi Tre News - Home |
| URL | `https://news.tuoitre.vn/home.rss` |
| 媒体 | Tuoi Tre News |
| 形式 | RSS 2.0 |
| フィード言語 | `vi-vn`（配信メタデータ値。記事本文は英語のため実記事の言語判定を別途行う） |
| タイムゾーン | 公開日時の `GMT+7` を解釈し、UTC で保存する |
| TTL | 配信値 20 分を尊重し、初期取得間隔は 30 分とする |
| 初期状態 | 有効 |
| 初回取得範囲 | フィードが現在返す全 item。記事本文ページの過去巡回は行わない |

- このフィードには `guid` がないため、正規化 URL を一次的な記事識別子として使用する
- `title`、`description`、`link`、`enclosure`、`author`、`pubDate` を取り込む
- Home フィードにはベトナム国外の記事も含まれるため、ベトナム関連度判定を必須とする

### FR-02 自動収集

- 有効なフィードを設定間隔で取得する。MVP の標準間隔は 30 分とする
- HTTP の ETag と Last-Modified が利用可能な場合は条件付きリクエストを使用する
- タイムアウト、一時障害、レート制限時は回数上限付きで再試行する
- 1 フィードの失敗が他フィードの処理を妨げない
- 手動実行と、指定期間を対象にした安全な再処理ができる
- 実行ごとに開始・終了日時、対象件数、追加件数、更新件数、失敗内容を記録する

### FR-03 記事の正規化と保存

- RSS/Atom の title、link、guid/id、published/updated、description/content、author、画像 URL を取得可能な範囲で保存する
- URL の追跡用パラメータ除去、文字コード、日時、HTML の正規化を行う
- 原文と処理後データを区別して保持し、処理を再実行できるようにする
- 日時は UTC で保存し、画面では Asia/Ho_Chi_Minh を標準表示とする
- 公開日時がない記事は取得日時を代用し、その旨を識別できるようにする

### FR-04 重複排除

- 第一優先でフィード ID/guid、次に正規化 URL、最後にタイトル・本文の類似度で重複を判定する
- 同一記事の更新は新規追加ではなく既存記事の更新として扱う
- 異なる媒体が同じ出来事を報じた場合、記事は残したまま同一の出来事へ関連付ける
- 判定理由と類似度を記録し、運用者が統合・分割できる

### FR-05 対象地域・カテゴリ判定

- ベトナムの地名、組織、人名、設定キーワード等を使って関連度を判定する
- 関連度が基準未満の記事は日誌に掲載しないが、管理画面では確認できる
- カテゴリは少なくとも「政治・行政」「経済・ビジネス」「社会」「災害・気象」「国際関係」「文化・スポーツ」「その他」を持つ
- 自動判定結果を運用者が修正できる

### FR-06 重要度判定

- まず掲載対象を「対象・対象外」で判定し、対象となった出来事に 0〜100 の参考スコアと「高・中高・中・低」の表示区分を付ける
- 重要度とは別に「必須掲載」フラグを持つ。必須掲載に該当する出来事は、スコア、モデル確信度、通常の掲載閾値にかかわらず人間の確認キューへ必ず残し、承認後の日誌から省略しない
- 初期段階では自動スコアを公開可否へ直結させず、人間の評価を補助する参考値として扱う
- 判定には、媒体の信頼度、複数媒体での報道、影響範囲、継続性、カテゴリ、固有名詞、緊急性を利用できる
- 判定では出来事の確定度を優先する。「決定した」「施行した」「署名した」「発生した」など確定した事実を、「計画している」「検討している」「目標としている」など未確定の内容より原則として上位に評価する
- 確定度は記事中の付随的な行為ではなく、日誌の主対象について判定する。主対象が未発生の災害なら、避難命令が発令済みでも災害自体は未確定として扱う
- 未確定の出来事でも、国家規模、重大な危険、期限の切迫などがある場合は、根拠を示して例外的に上位評価できる
- 企業関連では投資額に加え、企業の国際的知名度、市場規模、産業・サプライチェーンへの影響力を評価する
- 台風については「上陸済み」を重要な歴史的事実として扱い、人的被害の有無にかかわらず「高」を基本とする。一方、接近予報・警戒段階は「低」を基本とする
- スコアだけでなく、閲覧者が理解できる短い判定根拠を表示する
- 生成 AI を利用する場合も、ルールベースの最低条件、出典参照、失敗時の代替処理を備える
- AI の出力だけで原文にない事実を追加しない。確信度が低い結果は非公開または要確認にする
- 運用者はレビュー時に「掲載するか」「重要度」「判断理由」「迷った点」を入力できる
- 運用者はレビュー時に必須掲載フラグを設定・解除でき、その理由を記録できる
- システムはレビュー結果を変更履歴として保存し、判断例を比較しながら基準案を対話的に更新できる
- 重要度ルールまたはプロンプトを変更する際はバージョンを付け、過去データで再評価できる

### FR-06A AI プロバイダー

- 日誌生成処理は共通インターフェースを介し、少なくとも外部 API とローカルモデルを切り替えられる
- プロバイダーごとにモデル名、接続先、タイムアウト、同時実行数を設定できる
- 外部 API の認証情報はサーバー側の環境変数または秘密情報ストアで管理する
- ローカルモデルは OpenAI 互換 API を第一候補とするが、技術選定時に実際の仕様を確認してから確定する
- プロバイダーが利用不能な場合、記事を失敗扱いで破棄せず「処理待ち」または「要確認」に戻せる
- AI 出力は、要約、分類、重要度提案、判断根拠を構造化された同一スキーマへ正規化する
- 同じ評価用記事セットに対して API とローカルモデルを比較し、出典整合性、判断一致率、処理時間、コストを記録する
- Codex App Server 相当の機能は、公式にサーバー利用、認証、利用枠、無人定期実行が確認できた場合のみ追加プロバイダー候補とする。確認完了までは本番依存しない

### FR-07 出来事の日誌化

- 同一出来事の記事群から、出来事タイトル、2〜4 文の要約、発生日、カテゴリ、重要度、根拠、主要な固有名詞、出典一覧を生成する
- 「出来事が起きた日」と「記事公開日」が異なる場合は両方を保持する
- 発生日を特定できない場合は記事公開日を使用し、推定日であることを表示する
- 要約は出典に基づき、断定できない情報はその状態が分かる表現にする
- 新しい記事により内容が変わった場合は更新履歴を残す
- 公開状態は「公開」「要確認」「非公開」を持つ
- 初期運用では自動生成後の状態を必ず「要確認」とし、運用者が承認したものだけを「公開」に変更する

### FR-07A 段階的な承認自動化

- Stage 0（初期）: 全件を人間が確認し、承認・却下・修正理由を記録する
- Stage 1（提案支援）: 過去の判断例をもとにシステムが重要度と掲載可否を提案するが、公開には人間の承認を要する
- Stage 2（限定自動化）: 合意済み条件を満たす低リスクな出来事のみ自動公開し、高重要度、低確信度、情報矛盾、未知カテゴリは人間確認へ送る
- Stage 3（拡大）: 評価指標と定期監査を満たしたカテゴリに限り自動公開範囲を広げる
- ステージ変更は運用者による明示操作とし、精度評価、対象カテゴリ、閾値、ロールバック条件を記録する
- 自動公開後もサンプル監査を継続し、誤り率が基準を超えた場合は直前のステージへ自動または手動で戻せる

### FR-08 カレンダー表示

- 初期表示は当月の月カレンダーとする
- 日ごとに公開中の出来事件数を表示し、重要度「高」の有無を視覚的に区別する
- 前月・翌月・今日へ移動できる
- 日付選択で、重要度順の出来事一覧を表示する
- URL で年月日と絞り込み条件を共有できる
- モバイル幅でも日付と出来事へ到達できる

### FR-09 検索・絞り込み

- キーワード、日付範囲、カテゴリ、重要度、媒体で絞り込める
- 条件は組み合わせ可能で、解除できる
- 検索結果から日誌詳細と出典へ移動できる
- 該当なしの場合は明確な空状態を表示する

### FR-10 出来事詳細

- タイトル、要約、発生日、カテゴリ、重要度と根拠、更新日時を表示する
- 関連記事ごとに媒体名、原文タイトル、公開日時、原文への外部リンクを表示する
- 外部リンクであることを明示し、安全な別タブ遷移を行う
- 推定・未確認の情報は確定情報と視覚的に区別する

### FR-11 管理・監査

- 管理操作は認証済み運用者だけが実行できる
- 記事、出来事、収集ジョブを状態別に確認できる
- 重要度、カテゴリ、発生日、要約、公開状態を手動修正できる
- 誰が、いつ、何を変更したかを監査ログに残す
- 自動処理を同じ入力・処理バージョンで追跡できる

## 8. データ要件

| エンティティ | 主な属性 |
|---|---|
| Feed | ID、名称、URL、媒体、言語、カテゴリ、取得間隔、有効状態、信頼度、取得状態 |
| FetchRun | ID、Feed ID、開始/終了日時、HTTP 状態、追加/更新/失敗件数、エラー |
| Article | ID、Feed ID、guid、原文/正規化 URL、原文/正規化タイトル、本文抜粋、公開/更新/取得日時、言語、原データ、処理状態 |
| Event | ID、タイトル、要約、発生日、日付種別、カテゴリ、重要度、重要度根拠、必須掲載フラグ、必須掲載理由、関連度、公開状態、処理バージョン |
| EventArticle | Event ID、Article ID、類似度、主要出典フラグ |
| EventRevision | Event ID、変更前後、変更者、変更日時、変更理由 |
| ProcessingLog | 対象 ID、処理種別、入力版、モデル/ルール版、結果、エラー、処理日時 |

保持方針:

- 記事と出来事は、運用者が削除しない限り保持する
- 収集ログは最低 90 日、監査ログは最低 1 年保持する
- RSS 原データは再現性のため最低 30 日保持する
- 将来、保持期間を設定で変更できる構造にする

## 9. 非機能要件

### NFR-01 性能・可用性

- 月表示 API の 95 パーセンタイル応答時間は 1 秒以内を目標とする
- 1,000 フィード、累計 100 万記事まで、収集処理と閲覧処理を分離して拡張可能な設計とする
- 収集中でも既存の日誌を閲覧できる
- 失敗したジョブは再実行しても二重登録を起こさない

### NFR-02 セキュリティ

- 管理機能は認証・認可を必須とする
- RSS URL は SSRF 対策として許可プロトコル、DNS 解決先、リダイレクト回数、プライベート IP 宛通信を検証する
- RSS 内の HTML はサニタイズし、スクリプトを実行しない
- 秘密情報はソースコードやクライアントへ含めない
- 依存ライブラリとコンテナを定期的に脆弱性検査できること

### NFR-03 信頼性・観測性

- 収集件数、遅延、失敗率、未処理件数、AI/分類失敗率を記録する
- 連続失敗、日次処理未完了、未処理急増を通知可能にする
- 構造化ログにトレース可能なジョブ ID と対象 ID を含める
- DB の定期バックアップと復元手順を用意する

### NFR-04 アクセシビリティ・操作性

- キーボードのみで月移動、日付選択、詳細閲覧ができる
- 色だけに依存して重要度を表現しない
- 日付、リンク、見出しには読み上げ可能なラベルを付ける
- WCAG 2.2 AA を目標とする

### NFR-05 著作権・透明性

- 原則として RSS で提供された範囲と短い抜粋のみ保存・表示し、全文転載を前提にしない
- 媒体名と原文 URL を常に表示する
- 自動生成された要約であること、誤りの可能性、更新日時を明示する
- 各配信元の利用規約、robots 方針、取得頻度を確認できる台帳を持つ

### NFR-06 AI の可搬性・コスト管理

- ビジネスロジックを特定モデルのレスポンス形式へ直接依存させない
- プロンプト、入力、構造化出力、処理時間、トークン数または推定計算量、エラーを記録する
- 外部 API には日次・月次の利用上限と、上限到達時の停止またはローカル切替を設定できる
- ローカル処理ではモデル、量子化方式、必要メモリ、実行時間を再現可能な形で記録する
- 記事本文や個人情報を外部 API へ送る場合のデータ範囲を設定可能にする

## 10. 画面要件

### 10.1 カレンダー画面

- ヘッダー: サービス名、今日へ移動、検索、管理画面への導線（権限保有者のみ）
- 月ナビゲーション: 前月、年月、翌月
- カレンダー: 日付、公開出来事件数、最高重要度、カテゴリの要約
- 日別欄: 選択日の出来事タイトル、重要度、カテゴリ、要約冒頭
- 絞り込み: キーワード、カテゴリ、重要度、媒体

### 10.2 出来事詳細画面

- 出来事情報、重要度根拠、推定表示、関連出典、更新履歴
- 原文へ移動する明確な導線

### 10.3 管理画面

- フィード一覧・編集・接続テスト
- 収集ジョブ一覧と再実行
- 要確認記事・出来事のレビューキュー
- 出来事の修正、統合、分割、非表示

## 11. 処理フロー

1. スケジューラが取得対象フィードを選ぶ
2. RSS/Atom を取得し、実行ログを作る
3. 記事を正規化し、記事単位の重複を排除する
4. ベトナム関連度を判定し、対象外を除外する
5. 類似記事を既存または新規の出来事へ関連付ける
6. 出来事の発生日、カテゴリ、要約、重要度と根拠を生成する
7. 品質基準を満たす出来事を公開し、それ以外を要確認にする
8. カレンダー API/画面へ反映する
9. 失敗や異常値を記録し、再試行または運用者確認へ回す

## 12. 受け入れ条件

### AC-01 フィード登録と取得

- 正常な RSS/Atom URL を登録すると接続テストが成功し、有効化できる
- 無効な形式、重複 URL、プライベートネットワーク宛 URL は拒否される
- 有効化したフィードの新着が、設定した取得間隔後に記事として保存される
- `https://news.tuoitre.vn/home.rss` を取得し、RSS 2.0 の channel と item を解析できる
- `guid` がない item を正規化 URL で再取得しても重複登録しない
- `GMT+7` の `pubDate` を正しく UTC に変換し、画面ではベトナム時間で復元表示できる

### AC-02 冪等性と障害耐性

- 同じフィードを複数回取得しても同一記事が増殖しない
- 1 件のフィードがタイムアウトしても他のフィードは処理される
- 失敗理由と再実行結果を管理画面で確認できる

### AC-03 日誌生成

- 同じ出来事を扱う複数記事が、個別の出典を保ったまま 1 件の出来事にまとまる
- 公開された出来事には、要約、日付、カテゴリ、重要度、根拠、1 件以上の出典がある
- 出典にない固有の事実を要約へ追加しない
- 発生日が推定の場合、その旨が表示される

### AC-04 カレンダー閲覧

- 月を移動し、出来事のある日を識別できる
- 日付選択で、その日の公開出来事が重要度順に表示される
- 検索・絞り込み条件が URL に保持され、再読み込み後も復元される
- スマートフォン幅とキーボード操作で主要導線を利用できる

### AC-05 管理操作

- 未認証者は管理 API と管理画面を利用できない
- 運用者の修正内容が閲覧画面へ反映され、監査ログに記録される
- 出来事の統合・分割後も、関連する記事を失わない

### AC-06 AI 切替と人間承認

- 同一の入力契約で外部 API とローカルモデルを切り替え、同一形式の処理結果を保存できる
- AI が停止または不正な出力を返しても、元記事を失わず再処理できる
- 初期運用では人間が承認していない出来事が公開状態にならない
- 人間による重要度、掲載可否、判断理由が、使用したルール・モデルのバージョンとともに保存される
- 自動化ステージを戻すと、以降の対象記事が再び人間確認へ送られる
- 必須掲載候補は通常の掲載閾値で除外されず、人間の確認キューに表示される

## 13. 実装方針と段階

### Phase 0: 技術選定と外部仕様の確認

### Phase 4: 管理・運用と総合検証

## 16. 重要度基準の対話ログ

重要度基準は、実際の記事例に対する運用者の判断から作成する。この章の個別評価を正解データとして保存し、ルール、プロンプト、評価用データセットへ反映する。少数例から推定した傾向は「仮説」とし、追加評価で一貫性を確認するまで確定条件にしない。

### 16.1 第1回評価

| ID | 出来事 | 運用者評価 | 現時点の解釈 |
|---|---|---|---|
| IMP-001 | ベトナムの上半期 GDP 成長率が 8.18% | 高 | 国全体の経済状況を示す主要指標 |
| IMP-002 | ホーチミン市―ロンタイン空港を結ぶ約142億ドルの地下鉄計画 | 中 | 大規模で長期的な影響はあるが、現時点では計画段階 |
| IMP-003 | ベトナムとインドネシアが2028年の貿易額180億ドルを目標に行動計画へ署名 | 高 | 国家間の正式合意で、外交・経済の両面に影響 |
| IMP-004 | ホーチミン市警が観光客のひったくり容疑者を3時間で逮捕 | 対象外 | 個別事件であり、社会全体への波及が限定的 |
| IMP-005 | アルゼンチンがサッカーW杯決勝進出 | 対象外 | ベトナムとの直接的な関係がない |

### 16.2 第1回評価から得た判断仮説

1. ベトナムとの直接的な関係が、日誌掲載の必須条件である
2. GDP など国全体を表す主要経済指標は「高」になりやすい
3. 国家間の正式な合意は、経済的・外交的影響が見込まれる場合「高」になりやすい
4. 大規模インフラでも、計画段階では自動的に「高」とせず「中」を基本候補とする
5. 単発の犯罪・事故は、被害規模や制度変更など広い波及がなければ対象外とする
6. 金額や人数の大きさだけで決めず、確定度、影響範囲、継続性を合わせて評価する

第2回評価後、仮説3の「国家間の正式な合意」と、単なる目標・検討を明確に分ける。記事見出しに目標値が含まれていても、署名済みの行動計画など確定した行為が主題なら、その確定した行為を基準に評価する。

### 16.3 暫定判定順序

1. ベトナムと直接関係するかを判定し、関係がなければ対象外とする
2. 個別的・局所的な出来事で、広い波及がなければ対象外とする
3. 国家規模の指標、制度、正式な国際合意を「高」候補にする
4. 大規模だが計画段階、または影響が地域・業界に限られるものを「中」候補にする
5. 出来事の確定度と影響範囲を確認し、最終評価を人間へ提示する

| ID | 出来事 | 運用者評価 | 現時点の解釈 |
|---|---|---|---|
| IMP-006 | ベトナムのコーヒー輸出企業が EU 規制への対応準備を進めている | 中 | 主要輸出産業への影響はあるが、準備中であり確定した変化ではない |
| IMP-007 | ベトナムの航空便の約4分の1、6,200便以上が1か月で遅延 | 中 | 発生済みで影響は広いが、単月の運航実績としては中程度 |
| IMP-008 | ダナンの一自治体が、16歳未満の SNS・携帯電話利用規則を国内で初めて導入 | 高 | 地域限定でも、国内初の規則が実際に導入されたという制度的な先例 |
| IMP-009 | ホーチミン市の無料デジタル医療拠点が開設 | 中 | 実施済みだが、現時点の対象地域・利用者が限定的 |
| IMP-010 | フーコックの観光船転覆で15人が死亡し、業界団体が安全制度の見直しを要求 | 中高 | 重大な被害が発生済みで制度的波及も見込まれるが、制度変更自体は未決定 |

### 16.5 第2回評価で確定した原則

1. 未確定の計画・検討・目標より、決定・導入・署名・施行・発生済みの事実を優先する
2. 「対象外」は重要度の最低段階ではなく、ベトナムとの関連性や日誌対象性を表す別判定とする
3. 掲載対象の重要度は「高・中高・中・低」の4段階とする
4. 地域限定の出来事でも、国内初の制度や他地域へ波及し得る先例は「高」になり得る
5. 発生済みでも、影響が地域や一施設に限られる出来事は「中」が基本候補となる
6. 死者を伴う重大事故は個別事件として一律除外せず、被害規模と制度的波及を考慮する
7. 重大事故後の制度見直し要求は、事故の確定性と要求の未確定性を分けて評価する

### 16.6 暫定重要度マトリクス

| 影響範囲・制度的波及 | 確定済み | 未確定・計画段階 |
|---|---|---|
| 国家規模、国家間、国内初の重要制度 | 高候補 | 中高〜中候補 |
| 複数地域、主要産業、重大事故 | 中高〜高候補 | 中〜中高候補 |
| 一地域、一施設、限定された業界 | 中候補 | 低〜中候補 |
| 個人的・単発で広い波及なし | 低または対象外 | 対象外候補 |

この表は機械的な決定表ではない。媒体の信頼度、継続性、被害規模、前例性を加味し、人間の評価結果で更新する。

| ID | 出来事 | 運用者評価 | 現時点の解釈 |
|---|---|---|---|
| IMP-011 | 政府が全国最低賃金の引き上げを正式決定 | 高 | 全国に及ぶ制度変更が正式決定済み |
| IMP-012 | 経済団体が最低賃金の引き上げを政府へ提案 | 中 | 全国的な論点だが、提案段階で未決定 |
| IMP-013 | 大手企業がベトナム工場への10億ドル投資を正式決定 | 高（条件付き） | Intel、Tesla級の国際的影響力を持つ企業による正式決定なら高 |
| IMP-014 | 大手企業が同投資を検討中と表明 | 低 | 投資規模が大きくても検討段階で未確定 |
| IMP-015 | 台風が上陸し、死者なし・一部地域で停電 | 高 | 人的被害がなくても、台風上陸という自然災害の発生事実を重視 |
| IMP-016 | 強い台風が接近中で、政府が大規模避難命令を発令 | 低 | 行政対応は決定済みでも、主対象である台風被害は未発生 |

### 16.8 第3回評価で確定した原則

1. 国民・労働者へ広く影響する全国制度の正式決定は「高」とする
2. 全国規模の制度論点でも、業界団体等による提案段階は「中」を基本とする
3. 企業投資は、正式決定、投資規模、企業の国際的影響力を組み合わせて評価する
4. Intel、Tesla級の国際的知名度・産業影響力を持つ企業による大型投資の正式決定は「高」候補とする
5. 金額が大きくても、投資検討または関心表明だけなら「低」を基本とする
6. 台風の実際の上陸は、死者や大規模被害がなくても「高」とする
7. 災害の予報、接近、警戒段階は、大規模な避難命令が出ていても「低」を基本とする
8. 確定度は記事内の一部の行為ではなく、日誌の主対象となる出来事について評価する。例えば避難命令が確定済みでも、主対象が将来の台風被害なら未確定として扱う

### 16.9 確定度を判定する対象

記事には、予測、行政対応、実際の出来事が同時に含まれることがある。重要度判定前に「この日誌で記録する主対象」を一つ定め、その主対象について確定度を判定する。

| 記事内容 | 日誌の主対象 | 確定度の扱い |
|---|---|---|
| 台風接近予報と避難命令 | 将来の台風上陸・被害 | 未確定 |
| 台風上陸と停電発生 | 台風上陸・実被害 | 確定済み |
| 投資検討の表明 | 投資実行 | 未確定 |
| 投資契約または取締役会決定 | 投資決定 | 確定済み |
| 制度改正の提案 | 制度変更 | 未確定 |
| 政府による正式決定 | 制度変更 | 確定済み |

ニュース速報として予報や警戒情報を表示する機能を将来設ける場合は、歴史的な日誌の重要度とは別に「緊急性」を持たせる。現時点のMVPでは緊急性通知を日誌重要度へ混在させない。

1. 自然現象が発生済みであっても、それだけで一律に「高」とせず、現象の強度、被害、避難者数、影響範囲を組み合わせる
2. 熱帯低気圧の上陸と限定的な停電は「中」、小規模・無被害の地震は「低」とする
3. 死者数だけを災害重要度の中心指標にしない。数千人規模の避難など生活への広範な影響があれば「高」になり得る
4. 台風上陸は前回評価に従い「高」を基本とするが、他の自然現象は種類と実影響に応じて評価する
5. Samsung級の国際企業によるベトナムへの工場投資の正式決定は「高」とする
6. ベトナム国内の有力企業による大型工場の正式決定も「高」とする
7. 企業投資は金額だけで評価しない。企業の実体、信頼性、国内外での影響力、雇用、産業上の重要性を確認する
8. 無名企業による大型投資の発表は、正式発表済みでも「低」を基本とし、実行能力や波及効果の裏付けが得られた場合に再評価する

### 16.12 企業影響力の暫定評価項目

「大手」「有力」「無名」をモデルの印象だけで判定しない。利用可能な出典に基づき、次の項目を確認する。

- ベトナムまたは世界市場での売上・市場シェア
- ベトナム国内の既存雇用者数、拠点数、サプライチェーンへの影響
- 半導体、エネルギー、通信、交通など戦略産業との関係
- 上場、監査済み情報、過去の投資実績など企業実体の確認可能性
- 投資契約、認可、取締役会決議、着工など実行確度を示す証拠
- 想定雇用、地域経済、輸出、生産能力への具体的影響

6. 大規模な反政府デモの実発生は「必須掲載」とし、重要度は「高」とする
7. 著名な政府高官の正式逮捕は「高」とする。捜査、疑惑、告発だけの場合は確定度を下げて別途評価する

### 16.15 必須掲載の扱い

- 必須掲載は重要度の5段階目ではなく、日誌から欠落させないための独立した真偽値とする
- 必須掲載候補はAIが自動公開せず、初期運用では必ず人間確認へ送る
- 必須掲載の判定根拠、設定者、設定日時、解除履歴を監査ログへ保存する
- 同じ出来事の続報を無制限に必須掲載せず、一つの出来事へ統合した上で重要な転換点を改訂履歴として残す
- 必須掲載条件は、後述の第6回評価で追加された国家体制・指導部・武力衝突・大規模集団行動を含む

### 16.26 初期判定基準の完成状態

第1回から第9回まで、合計57件の評価例を収集した。MVPではこれらを初期評価データセットとして固定し、ルールベース判定、外部API、ローカルモデルの比較に使用する。

- 必須掲載、対象外、重要度4段階を区別できること
- 57件すべてについて、運用者評価とモデル提案の一致・不一致を記録すること
- 一致率だけでなく、必須掲載の見逃しと対象外記事の誤掲載を個別に測定すること
- 基準変更時に全57件を再評価し、前バージョンとの差分を提示すること
- この評価セットを学習へ直接使用する場合も、同じデータだけで性能評価を行わず、別の検証セットを用意すること
